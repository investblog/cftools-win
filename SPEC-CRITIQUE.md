# Critique of `SPEC.md`

Этот файл фиксирует, что в текущей спецификации хорошо как стартовая рамка, но недостаточно определено для уверенной реализации без дополнительных договорённостей.

Цель документа: не переписать `SPEC.md`, а перечислить, что нужно добавить, уточнить или исправить, чтобы спецификация стала инженерно пригодной.

## 1. Auth-модель слишком узкая и рискованная

### Что сейчас

`SPEC.md` описывает только `Email + Global API Key` и хранение секрета в Windows Credential Manager.

### Проблема

Для desktop bulk-tool это слишком широкие полномочия и плохой security baseline. Global API Key обычно хуже API Token по принципу минимальных привилегий, аудиту и ротации.

### Что добавить/исправить

- Явно определить, поддерживает ли MVP только Global API Key или также API Token.
- Если только Global API Key, написать почему это допустимо для MVP и какие риски принимаются.
- Зафиксировать минимально необходимые права для каждого сценария:
  - verify credentials
  - list accounts
  - list zones
  - create zone
  - purge cache
- Описать поведение UI при insufficient permissions.
- Добавить migration path: как позже перейти с Global Key на Token без переделки всей auth-модели.

## 2. Противоречие между "без persistence" и batch UX

### Что сейчас

Спецификация говорит, что в MVP persistence/ledger не нужен, но одновременно требует:

- progress
- pause
- cancel
- retry failed
- ETA
- массовые операции

### Проблема

Это уже не просто разовый вызов API, а полноценный batch workflow. Без минимального журнала или хотя бы чётко описанного ephemeral-поведения неясно, что происходит:

- при падении приложения
- при перезапуске ОС
- при временной сетевой ошибке посередине batch
- если пользователь хочет понять, что уже обработалось

### Что добавить/исправить

- Явно выбрать модель MVP:
  - batch полностью in-memory и теряется после закрытия приложения
  - или есть минимальная persistence для текущего batch/result log
- Если batch эфемерный, прямо прописать это в UX.
- Определить, можно ли восстановить результаты после рестарта.
- Определить, где хранится итог batch:
  - только на экране
  - в лог-файле
  - в экспортируемом отчёте
- Уточнить semantics `Retry Failed`:
  - повторяет только retryable ошибки
  - или любые failed-задачи

## 3. Недостаточно строгий контракт у `RequestPool`

### Что сейчас

Есть общие параметры concurrency/retry/backoff, но поведение сервиса описано на высоком уровне.

### Проблема

Для batch-операций это центральный компонент, а сейчас его контракт слишком расплывчатый. Из-за этого реализация может получиться несовместимой с ожиданиями UI.

### Что добавить/исправить

- Зафиксировать, принимает ли каждая операция `CancellationToken`.
- Разделить поведение `Pause`, `Resume`, `Cancel`, `Clear`.
- Определить, что считается "paused":
  - новые задачи не стартуют
  - уже выполняющиеся завершаются
- Определить, что считается "cancelled":
  - новые задачи не стартуют
  - активные запросы отменяются или дожидаются завершения
- Описать, как считаются `processed`, `success`, `failed`, `skipped`.
- Зафиксировать пределы:
  - max queue size
  - max total retries per task
  - max batch duration, если нужен
- Уточнить retry policy:
  - retry только на `429`
  - retry на `5xx`
  - retry на timeouts/network disconnect
  - нет retry на validation/auth/dependency errors

## 4. `DomainParser` слишком эвристический

### Что сейчас

Парсер описан как почти 1:1 порт логики из расширения, с regex, IDN-конверсией и special SLD rules.

### Проблема

Определение "root domain" без Public Suffix List ненадёжно. Эвристики вроде `co.uk`, `com.br` не масштабируются и быстро приводят к спорным кейсам.

### Что добавить/исправить

- Явно определить, нужен ли строгий registrable domain parsing или best-effort extraction.
- Если нужен строгий root-domain parsing, описать источник правил:
  - PSL
  - собственный ограниченный список
- Уточнить, что считается валидным входом:
  - домены по одному на строке
  - домены внутри произвольного текста
  - URL, email, mixed content
- Добавить таблицу edge-cases:
  - `www.example.com`
  - `sub.example.co.uk`
  - Unicode IDN
  - повторы в разном регистре
  - trailing dot
  - URL с path/schema
- Чётко разделить:
  - parser result
  - preflight result
  - окончательное решение API

## 5. Не хватает state-моделей для batch и preflight

### Что сейчас

Спецификация описывает пользовательские состояния вроде `will-create`, `exists`, `invalid`, `duplicate`, но типы данных в разделе моделей в основном ограничены API DTO.

### Проблема

Основная сложность этого приложения не в DTO Cloudflare, а в orchestration state. Если state-модели не определены в спецификации, логика расползётся по ViewModel и UI.

### Что добавить/исправить

- Добавить отдельные модели/enum для UI и batch-логики:
  - `PreflightStatus`
  - `TaskStatus`
  - `BatchStatus`
  - `OperationKind`
- Добавить сущности:
  - `TaskEntry`
  - `BatchSummary`
  - `PreflightEntry`
  - `OperationResult`
- Определить минимальный набор полей:
  - input
  - normalized input
  - status
  - error code
  - user-facing message
  - retryable flag
  - timestamps

## 6. Архитектурно слишком сильный акцент на "порт 1:1"

### Что сейчас

Много мест описаны как прямой порт из browser extension.

### Проблема

Это полезно как источник логики, но не как архитектурный принцип. Desktop app имеет другие границы и не обязан повторять extension-архитектуру.

### Что добавить/исправить

- Явно отделить:
  - API client
  - domain services
  - batch orchestrator
  - UI state/ViewModel layer
- Зафиксировать, какие части реально должны быть 1:1 по поведению с расширением.
- Зафиксировать, какие части можно адаптировать под desktop runtime.
- Не использовать формулировку "порт 1:1" там, где речь идёт скорее о заимствовании идей, а не точного контракта.

## 7. Не определены нефункциональные требования

### Что сейчас

Документ в основном покрывает функциональность и структуру проекта.

### Проблема

Без нефункциональных требований трудно принимать инженерные решения по логированию, UX, памяти, timeout policy и тестированию.

### Что добавить/исправить

- Добавить отдельный раздел `Non-Functional Requirements`.
- Зафиксировать:
  - ожидаемый размер batch
  - допустимое время отклика UI
  - максимальную параллельность
  - требования к памяти
  - timeout policy
  - offline/network flap behavior
  - logging policy
  - redaction policy для секретов
- Прописать, что нельзя писать в логи:
  - API keys
  - auth headers
  - чувствительные payloads

## 8. Не хватает раздела про observability и диагностику

### Что сейчас

Ошибки и нормализация описаны, но эксплуатационная диагностика не описана.

### Проблема

Для bulk-инструмента диагностика очень важна. Иначе пользователь увидит только "failed", а разработчик не поймёт, почему.

### Что добавить/исправить

- Описать уровни логирования:
  - info
  - warning
  - error
  - debug, если нужен
- Зафиксировать, где хранятся логи.
- Добавить correlation для batch/task/request.
- Определить, что пользователь видит в UI, а что остаётся только в диагностике.
- Уточнить, нужен ли экспорт отчёта по batch.

## 9. UI описан как список контролов, но мало описано поведение

### Что сейчас

UI sections в основном перечисляют элементы интерфейса.

### Проблема

Список контролов не задаёт UX-контракт. Многие ключевые вещи остаются неочевидными.

### Что добавить/исправить

- Для каждой страницы описать:
  - initial state
  - loading state
  - empty state
  - error state
  - completed state
- Уточнить блокировки UI:
  - можно ли менять account во время batch
  - можно ли редактировать input во время выполнения
  - что происходит при disconnect во время batch
- Описать confirm dialogs для опасных операций:
  - purge selected
  - disconnect/delete credentials
- Явно определить, нужна ли виртуализация списка для больших наборов зон/доменов.

## 10. Bulk Purge требует явных guardrails

### Что сейчас

Есть выбор зон и команда `Purge Selected`.

### Проблема

Это потенциально опасная массовая операция, но в спецификации почти нет защитных механизмов.

### Что добавить/исправить

- Добавить подтверждение перед purge.
- Описать, показывается ли число выбранных зон.
- Определить, разрешён ли purge для всех зон одним действием.
- Уточнить, нужен ли флаг/подтверждение для очень больших выборок.
- Описать, что делать с частичным успехом.

## 11. Cloudflare API section слишком общий

### Что сейчас

Перечислены endpoints, базовые request bodies и общий response format.

### Проблема

Для реализации batch-операций этого недостаточно. Не описаны ограничения и edge cases конкретных вызовов.

### Что добавить/исправить

- Для каждого endpoint описать:
  - expected success response
  - common errors
  - retryability
  - pagination behavior
- Для `GET /zones` явно указать:
  - `per_page`
  - как обрабатывается пагинация
  - лимиты на число страниц/запросов
- Для `POST /zones` описать known failure modes:
  - already exists
  - invalid zone
  - insufficient permissions
  - plan/subscription blockers
- Для purge описать, нужен ли rate limiting отдельно от create.

## 12. CI/CD section пока декларативный

### Что сейчас

CI/CD описан как укрупнённый список шагов.

### Проблема

Для WinUI/MSIX этого мало. Нет деталей, без которых pipeline нельзя воспроизвести.

### Что добавить/исправить

- Зафиксировать:
  - кто и чем подписывает MSIX
  - где хранится сертификат
  - как задаётся package identity/version
  - какие артефакты публикуются
  - как происходит release на tag
- Уточнить, какие проверки обязательны в CI:
  - restore
  - build
  - unit tests
  - packaging smoke test
- Если нужен self-hosted runner, описать зачем именно он обязателен, а что может идти на GitHub-hosted Windows runner.

## 13. Не хватает явного test strategy

### Что сейчас

Тестирование почти не специфицировано.

### Проблема

Без этого непонятно, как верифицировать корректность parser/request pool/error mapping до появления полноценного UI.

### Что добавить/исправить

- Добавить раздел `Testing Strategy`.
- Уточнить, какие компоненты покрываются unit-тестами в первую очередь:
  - `DomainParser`
  - `NormalizeError`
  - `RequestPool`
  - pagination logic
- Зафиксировать, какие вещи тестируются интеграционно.
- Определить, будут ли mock/stub ответы Cloudflare API.
- Добавить acceptance criteria для MVP.

## 14. Не определены границы MVP достаточно жёстко

### Что сейчас

Функции MVP перечислены, но не у всех зафиксированы исключения и consciously deferred cases.

### Проблема

Из-за этого MVP может бесконтрольно разрастись уже во время реализации.

### Что добавить/исправить

- Для каждой MVP-функции добавить раздел `Out of Scope`.
- Прямо исключить из MVP всё, что не планируется делать сейчас:
  - batch persistence
  - advanced filtering
  - export/import
  - token-based auth, если не берём
  - telemetry
  - localization
- Добавить явный `Definition of Done` для MVP.

## 15. Нужен короткий список конкретных правок в `SPEC.md`

### Рекомендуемый минимальный набор изменений

Если править быстро и по приоритету, я бы сначала внёс следующее:

1. Добавить разделы:
   - `Non-Functional Requirements`
   - `Testing Strategy`
   - `Out of Scope for MVP`
2. Расширить auth section:
   - типы credentials
   - required permissions
   - error behavior
3. Расширить batch sections:
   - lifecycle
   - cancel/pause semantics
   - partial success semantics
   - retry failed semantics
4. Переписать parser section:
   - чётко определить supported input
   - чётко определить root-domain semantics
5. Добавить state-модели:
   - preflight status
   - batch/task status
   - operation result
6. Детализировать CI/CD:
   - signing
   - packaging
   - required checks

## Итог

`SPEC.md` уже годится как продуктовая заготовка и архитектурный ориентир. Но в текущем виде он ещё недостаточно точен для бесконфликтной реализации.

Главные пробелы:

- auth и security assumptions
- batch lifecycle и durability
- parser semantics
- state-модели
- нефункциональные требования
- testing strategy
- реалистичный CI/CD контракт

Если нужен следующий шаг, логичнее всего не просто "дополнить по месту", а сделать вторую редакцию `SPEC.md` с более строгой структурой разделов и явными engineering constraints.
